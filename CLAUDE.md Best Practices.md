# CLAUDE.md Best Practices

CLAUDE.md (and AGENTS.md for [[OpenAI Codex]] / [[OpenCode]]) is the project instruction file that shapes every interaction between an [[AI Coding Harnesses|AI coding agent]] and your codebase. A well-written CLAUDE.md is the single highest-leverage investment you can make in AI-assisted development — it's [[Context Engineering]] applied to your project.

## What Goes in CLAUDE.md

### Essential Sections

```markdown
# Project Name

Brief description of what the project does (1-2 sentences).

## Tech Stack
- Language: TypeScript
- Framework: Next.js 14 (app router)
- Database: PostgreSQL via Prisma
- Testing: Vitest
- Package manager: pnpm

## Development Commands
- `pnpm dev` — Start dev server
- `pnpm test` — Run tests
- `pnpm test:watch` — Run tests in watch mode
- `pnpm lint` — Run ESLint
- `pnpm build` — Production build
- `pnpm db:migrate` — Run database migrations

## Architecture
- `src/app/` — Next.js app router pages and layouts
- `src/lib/` — Shared utilities and business logic
- `src/services/` — External service integrations
- `src/db/` — Prisma schema and migrations
- `src/types/` — Shared TypeScript types

## Conventions
- Use named exports, not default exports
- All API handlers must validate input with Zod
- Error responses follow the format: { error: string, code: string }
- Database queries go through service layer, never called from routes directly
- Tests go in `__tests__/` directories alongside source files

## Important Rules
- NEVER modify files in `src/generated/` — they are auto-generated by Prisma
- NEVER commit .env files
- Always run `pnpm lint` before committing
```

### Optional But Valuable Sections

```markdown
## Key Abstractions
- All payment providers implement the `PaymentAdapter` interface
- Background jobs use the `JobRunner` class in `src/jobs/base.ts`
- Auth is handled by middleware in `src/middleware/auth.ts`

## Common Pitfalls
- The `User` type from Prisma doesn't include relations by default — use `UserWithOrders`
- Redis connection must be awaited before use (see `src/lib/redis.ts`)
- Tests that touch the database must use the `setupTestDb()` helper

## Recent Decisions
- We migrated from Express to Next.js API routes (don't suggest Express patterns)
- We chose Prisma over Drizzle for the ORM (don't suggest switching)
```

## Writing Principles

### Be Specific and Verifiable

```markdown
# BAD — vague, unverifiable
- Write clean code
- Follow best practices
- Be careful with the database

# GOOD — specific, verifiable
- Use 2-space indentation (configured in .editorconfig)
- All functions must have explicit return types
- Database queries must use parameterized values (never string concatenation)
```

### Be Concise

Anthropic's research shows adherence drops beyond ~200 lines. Every line must earn its place.

```markdown
# BAD — wordy
When you are writing TypeScript code, please make sure that you always
use the `interface` keyword rather than the `type` keyword for defining
object shapes, as this is our team's convention and helps with
declaration merging and consistency across the codebase.

# GOOD — concise
- Use `interface` for object shapes, not `type`
```

### Tell It What TO Do, Not Just What NOT to Do

```markdown
# BAD — only negatives
- Don't use var
- Don't use any
- Don't use console.log

# GOOD — positive with rationale
- Use `const` by default, `let` when reassignment is needed
- Type everything explicitly (strict mode is on)
- Use the logger from `src/lib/logger.ts` instead of console.log
```

## Scoping and Organization

### Multi-File Strategy

For larger projects, split instructions across scoped files:

```
CLAUDE.md                          ← Project-wide rules
.claude/rules/api.md              ← Rules for src/api/** files only
.claude/rules/tests.md            ← Rules for **/__tests__/** files only
.claude/rules/database.md         ← Rules for src/db/** files only
```

Use YAML frontmatter for path scoping (Claude Code):
```markdown
---
paths:
  - "src/api/**"
  - "src/routes/**"
---
# API Rules
- All endpoints must be rate-limited
- Authentication is required except for /health and /auth/login
```

### Cross-Tool Compatibility

| Harness | File | Format |
|---|---|---|
| [[Claude Code]] | `CLAUDE.md`, `.claude/rules/*.md` | Markdown, YAML frontmatter |
| [[OpenAI Codex]] | `AGENTS.md` | Markdown, hierarchical discovery |
| [[OpenCode]] | `AGENTS.md` | Markdown |
| [[AI Coding Landscape\|Cursor]] | `.cursorrules` | Plain text |
| [[AI Coding Landscape\|Aider]] | `.aider.conf.yml` | YAML |

Tip: maintain both `CLAUDE.md` and `AGENTS.md` if your team uses multiple tools. They can be identical or tool-specific.

## Anti-Patterns

| Anti-Pattern | Problem |
|---|---|
| **Pasting entire config files** | Wastes tokens, not actionable context |
| **Vague aspirations** | "Write maintainable code" — not useful |
| **Excessive length** | >200 lines reduces adherence |
| **Outdated info** | Wrong commands or conventions confuse the agent |
| **Duplicating README** | CLAUDE.md is for the AI, not for human onboarding |
| **No build/test commands** | Agent can't verify its own changes |
| **Contradictory rules** | "Be thorough" + "Be concise" — which wins? |

## Generating CLAUDE.md

[[Claude Code]]'s `/init` command analyzes your project and generates a starting CLAUDE.md:

```bash
claude
> /init
```

This produces a reasonable first draft. Then:
1. **Review and edit** — The AI's analysis is usually 80% right
2. **Add team conventions** — Things the AI can't infer from code
3. **Add recent decisions** — "We chose X over Y because Z"
4. **Keep it updated** — When conventions change, update CLAUDE.md

## CLAUDE.md as Documentation

A well-written CLAUDE.md serves dual purpose:
- **For AI:** Context that shapes every agent interaction
- **For humans:** Onboarding guide for new developers

The best CLAUDE.md files are useful even to developers who don't use AI tools.
